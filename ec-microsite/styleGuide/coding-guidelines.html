Coding Standards and Guidelines

This document is a detailed guide to assist in the production, implementation and continuing development of the company website.  The document's primary goal is to set rules for code consistency and best practices.  This guide outlines the overall file structure, naming conventions, design and coding guidelines, styles and resources used in the construction of this site. To ensure design continuity and maintain the life span of this site, it is important that everybody involved in the development, maintenance and updating of this site read, understand and use these guidelines.   By adhering to best practices, we ensure optimized page loading, performance and maintainable code.

HTML

DOCTYPE Declaration

HTML5 will be used.  In order to conform to HTML5, every HTML page on the site must start with the following DOCTYPE declaration.

<!DOCTYPE html>

The DOCTYPE declaration indicates to the web browser the particular language you intend to use within the document and the level of adherence to the standard you intend to follow.   Proper usage of the DOCTYPE declaration can affect the display of the page, the interpretation of style sheets and the ability to validate html syntax.  Browsers’ modes vary greatly based on DOCTYPE.

Character Encoding

All pages should use the Unicode UTF-8 character set.

<meta http-equiv="Content-Type" content="text/html; charset=utf-8" />

Prevent IE Compatibility Mode

Internet Explorer includes a Compatibility View feature that displays websites that were created for older browsers as they were designed to appear.  Sometimes IE will switch to compatibility mode (Quirks mode) without you knowing.  Include the following attributes in the site <head> to prevent your site from defaulting to compatibility mode.  It will change the engine that is being used to first Chrome Frame, if the plugin is installed, and then to Edge (the highest supported document mode of the browser). 

<meta http-equiv="X-UA-Compatible" content="IE=edge,chrome=1">

Page Descriptions

Page descriptions should describe the content and purpose of the page in 1-3 short sentences.

<meta name="Description" content="Page summary goes here." />

Page Keywords

Page keywords are used to add synonyms and additional meaning to a page.  Keywords should not be used to add irrelevant or unrelated words to a page.

<meta name="keywords" content="keyword 1, keyword 2, keyword 3" />

Page Title

To encourage ease of navigation and good search engine ranking, all pages should contain a title, description and keywords.  These content elements should be created by the content editor and should not be left to the developer.
Page title should be simple, descriptive and distinct, avoiding marketing hyperbole.  For consistency, all titles should be in the format:

<title>[Site Name] – [Section] : [Page]</title>

General Markup Guidelines

Markup

The following are general guidelines for structuring your HTML markup.  Authors are reminded to always use markup which represents the semantics of the content in the document being created.

  • We recommend all indentation to be done via soft tabs (using the space character).  Hitting Tab in your text editor shall be equivalent to four spaces.   The reason for this is when multiple developers use different tab settings the code becomes difficult to read or print, which is why spaces are preferable to tabs. 
  • Make use of <DL> (definition lists) and <BLOCKQUOTE>, when appropriate.
  • Use label fields to label each form field, the [for] attribute should associate itself with the input field, so users can click the labels
  • Set the width of input fields using CSS width and not inline like the size attribute.

Paragraphs

Use actual <P> elements for paragraph delimiters as opposed to multiple <BR> tags.

Good:



<p>This is content within a paragraph</p>

<p>This is another paragraph with content</p>

Bad:

<p>This is content within a paragraph

<br><br>

This is another paragraph with content, but the break tags above is bad  markup</p>

Tables

  • Tables should not be used for page layout.  HTML tables are intended for tabular data only.  All layout and structure will be done in the CSS files.
  • Make use of THEAD, TBODY, and TH tags and Scope attribute when appropriate.

<table width="500" cellspacing="1" cellpadding="10" summary="This will describe the contents of this table">

  This is the Table Title
  </caption>

      <th scope="col">Column Head 2</th>
      <th scope="col">Column Head 3</th>
    </tr>

  <tbody>
    <tr>
      <th scope="row">Row Head 1</th>

      <td>&nbsp;</td>

Tags and Attributes

Always use lowercase text in your HTML tags and attributes.  All attributes for tags must have a value and be surrounded by double quotes.

Good:

<img src="image.gif" alt="picture"  width="50" height="50" />

Bad:

<IMG SRC="image.gif" ALT="picture" WIDTH=50 HEIGHT="  " />

Some attributes that did not require descriptive elements in HTML require them in XHTML.  For example the ‘nowrap’ attribute for <td nowrap> tags or the ‘selected’ attribute for <option selected>.   To make these XHTML compliant, you need to add a description of the attribute.  For example:  <td nowrap="nowrap">

Close all of Your Tags

All tags should have an opening and a closing tag.   As an example make sure all of your <p> tags are followed by a </p> tag at the end of the paragraph and all of your <li> tags are followed by a </li> at the end of the list item.

Self-closing Elements

A space is required before the forward slash of a self-closing element.

Good:

<br />
<img src="image.gif" alt="picture" width="50" height="50" />

Bad:

<br/>

<img src="image.gif" alt="picture" width="50" height="50"/>

Alternative Text for Images

Alternative text is required for every image element on the site.  Alt text must provide an adequate description of the content or function of the image.  As a rule of thumb, consider what you might say if you were reading the web page to someone over the telephone.  You do not need to include the words "image of" or "graphic of."  For images that contain words or letters - use alternate text that includes the same words or letters.

<img src="images/graphic.gif" alt="Image Description Here" width="100" height="100" />

Links

The text within links should be unique and describe the destination of the link.  Links saying “Click Here” or “More” should be avoided.

<a href="news.html">More News Stories</a>

You can provide additional, non-critical link information in the form of a tool-tip by adding a title attribute.

<a href="news.html" title="News Archive">More News Stories</a>

Links to downloadable documents

Links to downloadable documents – such as PDF’s and Word documents – should contain the document size, format and the fact that they are downloads rather than links to another page.

<a href="minutes.pdf">Download the Minutes in PDF format (25KB)</a>

Image formats

There are four main image formats that should be used, detailed here:
JPEG -  This will cover all photography-based images, such as page promo images, or anything with a very high color count.

PNG24  - This format outputs high-color count imagery and fully supports graded opacity by pixel.  Relatively, it's quite a heavy format as far as kilobyte weight.  Note that IE6 has an issue rendering the transparent pixels, where PNG24's appear to have a gray or light-blue background.  In many cases, you should use a GIF fallback for IE6, in place of a PNG24.  You can apply a PNGfix but they are very slow and expensive on page weight and performance, so it's best to avoid using them.

PNG8 -  Offers 256 colors.  PNG’s offer good color and are sometimes a lot more compressible than GIF.  This format allows graded opacity in nearly all browsers, but in IE6.

GIF 89a. - GIF 89a offers the flexibility of transparency and wide browser support, but the constraints of no graded opacity or a color depth above 256.  The file size is comparably very small.  All low-color count imagery such as illustrations, icons and thumbnails should be done in PNG8.  It's the most size-efficient of these four.

Classes vs. IDs

You should only give elements an ID attribute if they are unique.  They should be applied to that element only and nothing else.  Classes can be applied to multiple elements that share the same style properties.  Things that should look and work in the same way can have the same class name.

<ul id="nav-bar">
  <li class="item">Category 1</li>
  <li class="item">Category 2</li>
  <li class="item">Category 3</li>
</ul>

CSS Guidelines

CSS describes all presentation aspects of the page via a description of its visual properties.  These properties are mixed and matched via identifiers to control the page's layout and visual characteristics through the layering of style rules.

Inline style attributes should be avoided (style="...").  Please use external style sheets or style tag declarations in the head section.  Do not put style tag declarations outside of the head section.

Linked Style Sheets

Add CSS through external files, minimizing the amount of code needed for each page of the site.  It should always be in the HEAD of the document.

Use the <link> tag to include the stylesheet

Good:

<link rel="stylesheet" type="text/css" href="myStylesheet.css" />

Inline Styles

Don't use inline styles in the document.  It's harder to track down these style rules.  It also adds to the page weight and code clutter.

Bad: 

<p style="font-size: 12px; color: #FFFFFF">No Inline Styles Please</p>

Browser Workarounds

Hacks are dangerous, since they are based on non-standard exploits, you can't predict how they are going to behave in future browsers.  The tool of choice for fighting IE problems is the conditional stylesheets.   When implementing browser-specific workarounds, they should be kept in separate files if possible.   For IE browsers, you can selectively target them in your HTML:

Target IE 7 ONLY

<!--[if IE 7]>
<link rel="stylesheet" type="text/css" href="ie7.css">
<![endif]-->

Target IE 7 and LOWER

<!--[if lte IE 7]>
<link rel="stylesheet" type="text/css" href="ie7-and-down.css" />
<![endif]-->

Target IE 8 ONLY

<!--[if IE 8]>
<link rel="stylesheet" type="text/css" href="ie8.css">
<![endif]-->

Target IE 9 and LOWER

<!--[if lte IE 9]>
<link rel="stylesheet" type="text/css" href="ie9-and-down.css" />
<![endif]-->

ID and Class Naming Conventions

ID and Class names will use compound words or phrases in which the elements are joined by the – hyphen character.  All lower case words will be used.

It is always preferable to name something, be it an ID or a class, by the nature of what it is rather than by where it is or what it looks like.  For instance, a class name of big-blue-text for a special note on a page is quite meaningless if it has been changed to have a small red text color.  Using a more intelligent and generic convention such as note-text is better because when the visual style changes it still makes sense.

Good:

note-text
tooltip
sidebar
external-link

Bad:

big-blue-text
curved-border-box
right-column
red-link

selector {
  property:  value;

Selectors

Write selectors that are optimized for speed.   Where possible, avoid expensive CSS selectors.   For example, avoid the * wildcard selector and don't qualify ID selectors (ex: div#myid) or class selectors (ex: table.style-02).   This is especially important with web applications where speed is paramount and there can be thousands of DOM elements.

Good:

#myid
.style-02

Bad:

div#myid
table.style-02

CSS Shorthand

The use of CSS shorthand will be used whenever possible.   Shorthand properties can be used to set several properties at once, in a single declaration, instead of using a separate declaration for each individual property.  This can save a lot of space in the CSS file.   This is not a requirement but a recommendation.  So you can type out all property value pairs if you are not comfortable with shorthand coding.

The following examples show the most common shorthand properties:
Background Property:

Good:

#selector {
  background: #fff  url(../images/graphic.png) repeat-x fixed left bottom;
}

Longhand is unnecessary:

#selector {
  background-color: #fff;
  background-image:  url(../images/graphic.png);
  background-repeat:  repeat-x;
  background-attachment:  fixed;
  background-position:  left bottom;
}

Font Property:

Good:

#selector {
  font: 13px/1.5 bold  italic serif;
}

Longhand is unnecessary:

#selector {
  font-size: 13px;
  line-height: 1.5;
  font-weight: bold;
  font-style: italic;
  font-family: serif;
}

Omit any of these commands from the background CSS shorthand property, and the browser will use the default values. If you leave out the background-position command then any background image will be place in the top-left of the container and then repeated both horizontally and vertically.

Lists

Good:

#selector {
  list-style: disc  outside url(image.gif); 
}

Longhand is unnecessary:

#selector {
 list-style: #fff;
 list-style-type: disc;
 list-style-position:  outside;
 list-style-image:  url(image.gif);
}

Margin

Good:

#selector {
  margin: 2px 1px 3px 4px;

}

Longhand is unnecessary:

#selector {
  margin-top: 2px;
  margin-right: 1px;
  margin-bottom: 3px;
  -left: 4px; 
}

Padding

#selector {
  padding: 0; (top,  bottom, right and left)
}

Longhand is unnecessary:

#selector {
  padding-top: 0;
  padding-right: 0;
  padding-bottom: 0;
  padding-left: 0;
}

Border

Good:

#selector {
  border: 1px black solid
}

Longhand is unnecessary:

#selector {
  border-width: 1px;
  border-color: black;
  border-style: solid;
}

CSS Formatting

To ease potential headaches for maintenance, it is recommended that all CSS must be written in a consistent manner. For one, all CSS selectors must be listed on their own line.  If multiple selectors are listed for a rule, each selector should be followed by a line break.  So as a general rule of thumb, if there is a comma in CSS rule, it should immediately be followed by a line break.  This way, we know that all text on a single line is part of the same selector.  Likewise, all property/value pairs must be on their own line with standard indentation (4 spaces), and end with a semicolon.  All property values must begin with a space (after the colon).  The closing brace must be on the same level of indentation as the selector that began it - flush left.

Good:

#selector-1 span,
#selector-2 span,
#selector-3 span {
  background: #fff;
  color: #000;
  font-size: 13px;
  font-weight: bold;
}

Bad:

#selector-1 span, #selector-2 span, #selector-3 span {
    background:#fff;  color:#000; font-size:13px; font-weight:bold; }

Links Styling

Default styles for links should be declared and different from the main text styling, and with differing styles for hover state.

Vendor-specific extensions

In CSS, keywords and property names beginning with - (dash) or _ (underscore) are reserved for vendor-specific extensions. 
If you must use extensions, you should use those that are closely related to equivalent CSS properties (be that CSS1, 2, or 3), so that you can switch to the standard property later on, and remove the extension when the browser implements the correct specification.  Note that prefixes are used by browsers to try out new experimental properties.  These extensions may be subject to change at the vendor’s whim, as they don’t form part of the CSS specifications.  So it is recommended that you avoid using them unless it’s absolutely necessary.
